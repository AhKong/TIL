## Docker Image 📸

- 도커에서 서비스 운영에 필요한 서버 프로그램, 소스 코드 및 라이브러리, 컴파일된 실행 파일을 묶는 형태 
- 즉, 컨테이너 생성 및 실행에 필요한 모든 파일과 설정값을 가지고 있음
- 컨테이너를 통해 더이상 의존성 파일을 컴파일 하거나 이것저것 설치를 할 필요 없는 상태의 파일을 의미
- 예를 들어 Oracle database는 mac os를 지원하지 않음 그렇지만 oracle db 이미지가 있다면 맥에서도 oracle db 이용 가능함 
 (이같은 경우는 사실 도커라는 존재도 모르고 학원 다닐때 오라클을 사용했어야 하는 상황이 있었음 그래서 그냥 맥에서 오라클 사용법을 검색하다가 우연히 사용하게 되었는데 이게 이런 원리인지는 당시에는 몰랐었다.)

- 도커 이미지의 용량은 작을 수도 클 수도 있음 하지만 팩트는 가상머신의 이미지에 비하면 굉장히 작은 용량이라는 점이다.
- 이미지는 상태 값을 가지지 않고 변하지 않는다. (Immutable)
- 하나의 이미지로 여러개의 컨테이너 생성 가능 
- 컨테이너가 삭제 되더라도 이미지는 변하지 않고 그대로 남아있다.
- 도커 이미지들은 Dockerhub를 통해 버전 관리 및 배포 가능
- 도커는 Dockerfile 이라는 파일로 이미지를 만든다.
- Dockerfile에는 소스와 함께 의존성 패키지 등 사용했던 설정 파일을 버전 관리하기 쉽도록 명시 되어 있습니다.
- 그렇기 때문에 Dockerfile 을 통해 누구나 이미지 생성과정을 확인 및 수정이 가능합니다!

## layer
- 기존 이미지에 추가적인 파일이 필요할 때 다시 다운 받지 않고 해당 파일을 추가하기 위한 개념
- 이미지는 여러 개의 읽기 전용 layer로 구성이 되고, 파일이 추가되면 새로운 layer가 생성된다.
- 도커는 여러개의 layer를 묶어서 하나의 파일시스템으로 이용가능하게 함
- 이런 면에서 이미지와 레이어는 같은 의미로 사용된다고 함.
- 추가적으로 Dockerhub 및 개인 저장소에서 이미지를 공유 할 때는 바뀐 부분(layer = image)만 주고 받는게 가능하다고 함.


## Docker container 

- `이미지를 실행한 상태`
- 응용프로그램의 종속성과 함께 응용프로그램 자체를 패키징하여 격리된 공간에서 프로세스를 동작시키는 기술 
- 컨테이너는 이미지 레이어에 읽기/쓰기 레이어를 추가하는 것으로 생성 및 실행이 된다. 따라서 여러개의 컨테이너를 생성해도 최소한의 용량만 사용되며, 바뀐 부분을 읽기/쓰기 layer에 적는다 ("???????? 이부분 질문할 수 있으면 질문하기!!@)
- 컨테이너를 종료 != 메모리에서 삭제 의 개념이 아니다.삭제를 하기 위해서는 명시적으로 삭제 해야한다. 즉, 종료가 되었다고 한들 언제든 다시 도커 명령어로 실행시키면 된다!
- 컨테이너를 삭제 == 컨테이너에서 생성한 모든 파일 삭제 
(ex oracle container 삭제 -> 그 안에 있는 모든 데이터 삭제)
- 한 서버는 여러개의 컨테이너 가져도 당연히 아무 문제 없다  `컨테이너는 각각 독립적으로 실행이 되기 때문`이다.
- 컨테이너는 커널 공간과 호스트 OS 자원(시스템 콜)을 공유한다.
